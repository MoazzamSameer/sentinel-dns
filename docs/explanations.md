# Plain-language explanations

The differentiator from the [research](RESEARCH.md): every block produces a plain-English reason a normal user can understand. Pi-hole says "blocked." Quad9 says "blocked for security." We say *why*.

This task delivers the generator. The next two tasks (SQLite query log, CLI `explain`) make it user-visible end-to-end. For now, every block gets an `explain` log line.

## Implementation

[`sentinel_dns/explanation.py`](../sentinel_dns/explanation.py) — ~140 lines.

Two dataclasses + one function:

```python
@dataclass(frozen=True)
class Reason:
    signal: str   # short stable code (for log fields, metrics)
    human: str    # one-sentence user-facing explanation

@dataclass(frozen=True)
class Explanation:
    qname: str
    decision: str          # "block" or "allow"
    reasons: list[Reason]
    human: str             # all reasons stitched together

def explain(qname: str, decision: Decision) -> Explanation: ...
```

The `human` string is **templated from structured reasons, not generated by an LLM**. Latency-safe (~10µs total), deterministic, no external dependency. The same templates serve both the log and the upcoming CLI `explain` command.

## What it covers

The block source comes first (it's the load-bearing reason):

| `block_source` | Signal code | Human template |
|---|---|---|
| `"blocklist"` | `blocklist` | "Listed on a known-malware threat feed (URLhaus)." |
| `"classifier"` | `lexical_classifier` | "Our ML model is N% confident this name looks malicious." |

Then heuristic color — included regardless of block source, because heuristics often fire alongside the load-bearing signal and add useful context:

| Heuristic signal | Sample human text |
|---|---|
| `abused_tld` | "Uses the .xyz TLD, frequently abused for short-lived attack domains." |
| `very_high_entropy` / `high_entropy` | "The name has very high character entropy, strongly suggesting it's algorithmically generated." |
| `very_long` / `long` | "Unusually long domain name (52 characters)." |
| `digit_heavy` | "Unusually digit-heavy for a typical domain name." |
| `many_dashes` | "Contains many dashes — common in typosquats and DGA-generated names." |
| `hex_label` | "Contains a long hex string, common in algorithmically-generated domains." |
| `consonant_run` | "Contains an unusual run of consonants, suggesting non-natural-language origin." |

Each `Reason` is independently emittable as either a signal code (machine-friendly) or a sentence (human-friendly). The `human` field on `Explanation` is just `" ".join(r.human for r in reasons)` — trivial concatenation today; smoothed with comma-separation and "and" is post-v0.1 polish.

## Wired into the BLOCK log line

The forwarder's existing `BLOCK` log gains a `signals=` field; a separate `explain` line follows with the human string. Two lines per block, cleanly separable.

```
INFO ... BLOCK qname=1ce6-route.fixionmunici9al.lat ml=0.9980 heur=0.300 ... source=classifier inline_us=482.4 signals=lexical_classifier,abused_tld
INFO ... explain qname=1ce6-route.fixionmunici9al.lat — Our ML model is 100% confident this name looks malicious. Uses the .lat TLD, frequently abused for short-lived attack domains.
```

`explain` lines only fire on actual blocks (not measurement-mode `score` lines, not allows). Avoids log spam.

## Design choices

- **Heuristic signals re-computed, not cached.** The `Decision` cache stores `heuristic_score` (a float) but not the structured `HeuristicReasons`. Recomputing is ~7µs (see Spike A microbench), much cheaper than blowing the cache size up by storing the structured form. The `explain()` function is the one place that needs the structured form, and it doesn't run on the hot path.
- **No LLM at query time.** Templated sentences are deterministic, fast, auditable, and don't leak query data to a third-party API. An LLM rewrite of the human string for variety / polish could happen offline at install time if we ever wanted that.
- **Allows produce no `explain` line.** Logging an explanation for every allow would be log spam (and a privacy footgun). The CLI `explain <domain>` command will let users opt in to checking individual allows.
- **`signal_codes` property on `Explanation`** for easy log/metric construction without re-parsing the human string.

## Caveats

1. **Reasons re-derived from qname only.** If we ever introduce signals that depend on context (e.g. per-client request rate), they won't be reconstructible from the cached `Decision`. Will need to extend the cache schema then.
2. **No typosquat detection yet.** The architecture's example explanation cited typosquatting (`g00gle-secure-login.xyz`); we don't currently have a typosquat heuristic. Once we do, it slots into `explain()` as another `Reason`.
3. **English only.** All template strings are hardcoded English. Internationalisation is post-v0.1.
4. **Templates are duplicated text.** Each signal's human string is a string literal in `explain()`. Fine at the current size (~10 signals); refactor to a registry/table if it grows past 20.
5. **No "confidence" gradation in the human text.** A 0.85 ML score and a 0.99 ML score both render as "N% confident," which is technically correct but doesn't communicate the qualitative difference. A future polish pass could add language gradation.
6. **No sources or links.** The blocklist explanation says "URLhaus" but doesn't link to the URLhaus listing for the specific domain. Could add for the CLI `explain` command but feels like log noise.

## What this unblocks

- The next task on the list — **SQLite query log** — has a structured form to persist (`Reason` + signal codes go straight into a relational schema).
- The task after that — **CLI `sentinel-dns explain <domain>`** — has a function it can call directly. Just `print(explain(qname, cached_decision).human)`.
- The "why blocked" differentiator from RESEARCH.md is now real, not a promise.
