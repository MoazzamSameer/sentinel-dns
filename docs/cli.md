# CLI

Until this task, `sentinel-dns` was a single-purpose console script that ran the forwarder and nothing else. Inspecting traffic meant either watching stdout or running ad-hoc `sqlite3` queries against the log file. Neither is something a normal user would do.

This task adds the first inspection subcommand — `sentinel-dns tail` — and the dispatcher infrastructure for the next ones.

## Layout

```
sentinel-dns                            → forwarder (default, backward compatible)
sentinel-dns --listen-port 5354 ...     → forwarder (still works exactly as before)
sentinel-dns tail [--flags...]          → SQLite query log streamer
sentinel-dns --help                     → top-level help (lists subcommands)
sentinel-dns tail --help                → tail's own argparse help
```

The dispatcher in [`sentinel_dns/cli.py`](../sentinel_dns/cli.py) is ~30 lines. The first positional arg picks the subcommand; if the first arg is missing or starts with `-`, control falls through to the existing `forwarder.main()`. This preserves the original entry-point behavior — `sentinel-dns --config foo.toml` keeps working without change.

The console script in `pyproject.toml` now points at `sentinel_dns.cli:main` instead of `sentinel_dns.forwarder:main`.

## `sentinel-dns tail`

[`sentinel_dns/tail_cmd.py`](../sentinel_dns/tail_cmd.py) — ~150 lines.

### What it does

Two modes against the SQLite query log:

1. **One-shot** (default): print the last N records and exit.
2. **Follow** (`-f`): print the last N then keep streaming new ones as the forwarder writes them.

Filters (combinable):
- `--decision allow|block` — only one type
- `--client <ip>` — one client
- `--qname-contains <substr>` — substring match
- `--min-ml-score <float>` — score threshold
- `--block-source blocklist|classifier` — distinguish where the block came from

Each block record gets a follow-up explanation line, generated by the same `explain()` function the forwarder uses for its `BLOCK` log line. The user-visible `tail` output is the same prose the forwarder logs to stdout for live blocks — single source of truth.

### Concurrency safety

The reader connects with `file:<path>?mode=ro` URI form. Combined with the forwarder's WAL journal mode, this lets the tail run while the forwarder writes — no lock contention, no waiting.

The `cache=hit` rows on the repeated `google.com`/`github.com` queries are visible — the cache works, and the tail makes that visible.

### Filter examples

```
# Just blocks from the last hour or so
sentinel-dns tail --decision block

# Just classifier-driven blocks (vs blocklist hits)
sentinel-dns tail --decision block --block-source classifier

# Anything 70%+ confidence — useful for tuning the threshold
sentinel-dns tail --min-ml-score 0.7

# What did the smart TV try to reach?
sentinel-dns tail --client 192.168.1.42 --qname-contains amazon -f
```

## Verification

Smoke test ran four cases against a forwarder writing to a fresh SQLite file:

| Case | Result |
|---|---|
| Top-level help | Prints usage + lists subcommands ✅ |
| Unknown subcommand (`sentinel-dns noisefilter`) | Errors with usage ✅ |
| `tail --help` | Shows tail's own argparse help ✅ |
| `tail -n 10` against 8 queries | All 8 shown chronologically with explanations on blocks ✅ |
| `tail --decision block` | Only the 2 block rows shown ✅ |
| `tail -f` while forwarder kept writing | New blocks streamed live within ~0.5s ✅ |

The follow-mode test specifically confirmed live streaming: started `tail -f --decision block`, then sent fresh queries, saw a new `BLOCK xxxxx.lat (.lat TLD, ML 93%)` record show up in the tail output without restarting.

## Caveats

1. **Default `--log-path` is `sentinel.db` in the current directory.** No XDG / `/var/lib` discovery. Pass `--log-path` explicitly if you keep the file somewhere else (most users will). Same posture as `--config` — explicit beats implicit.
2. **Polling, not native LISTEN/NOTIFY.** SQLite doesn't have first-class change notifications, so follow mode polls every `--poll-interval` seconds (default 0.5s). Means up to 0.5s lag in follow mode, which is fine for a human watching but explicit.
3. **No colors.** ANSI color output for blocks (red) vs allows (default) would be nice, especially with `BLOCK` highlighted; deferred to keep this PR small. If you want it, the right hook is `_format_row` and a `--color {auto,always,never}` flag.
4. **No JSON output mode.** Raw `sqlite3` CLI is the escape hatch for now; `--json` for piping into `jq`/scripts is post-v0.1 polish.
5. **`-n 0 -f` shows recent history.** If you want only new records, the workaround is `-n 0 -f` accepts but `last_id` starts at 0, so the first poll returns all rows. Documenting rather than fixing — common case (`-f` alone) works as expected.
6. **No live tail of stdout (forwarder logs).** This is `tail` for the SQLite query log specifically. `journalctl -fu sentinel-dns` or piping the forwarder's stdout is the answer for the other shape.
7. **Forwarder's internal stdout output is unchanged.** The `score`/`BLOCK`/`explain` log lines still print to the forwarder's stdout. `tail` is additive — a separate consumer of the same data.

## What this unblocks

- **`sentinel-dns explain <domain>`** (next task) reuses the same dispatcher pattern and the same SQLite reader. Should be a much smaller PR.
- The first sentence of the README quickstart now reads cleanly: *"Run `sentinel-dns --config sentinel-dns.toml` in one shell, then `sentinel-dns tail -f` in another to watch traffic."*
- Demo material is much easier to produce — open a tail in a terminal, query a few domains, the output is self-explanatory.
